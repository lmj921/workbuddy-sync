# Workflow：数据先行（Data-First SOP）

> **定位**：财经编辑的"第一道闸"。没有数据，就没有下笔权。
> **回答的问题**：这条稿子我动笔前，哪些数据必须先拉到手？
> **触发时机**：SKILL.md 主流程 Step 1.5——**拿到原材料之后、进入 evaluation 之前**。
> **硬规则锚点**：SKILL.md 硬规则第 10 条"数据先行，先查再写"。

---

## 🎯 核心理念

**搬运工看到热点 → 直接写**
**资深编辑看到热点 → 先查，再写**

财经稿件的全部公信力建立在**每一个数字都能追溯到权威来源**。只要有一个"大概"、一个"估计"、一个"记忆中是这个数"，整篇稿子的专业性就崩了。

**一句话铁律**：
> **写稿前 3 分钟的检索，能挡掉发稿后 3 天的打脸。**

---

## 🧱 三档数据检索（出稿前必跑）

按热点类型，至少命中其中 1 档；超级事件（evaluation ≥ 40）通常三档都要跑。

### 档位 1｜事件事实检索（WebSearch 路线）

**什么时候必跑**：
- 任何涉及"今天/今日/刚刚发生"的突发事件
- 涉及人物表态、官方公告、监管动作
- 涉及公司临时公告、财报日、发布会
- 涉及地缘政治、海外市场、跨境事件

**检索清单**（按优先级）：
| 优先级 | 信源类型 | 代表媒体 | 用途 |
|---|---|---|---|
| P0 | 官方 | 新华社、央视新闻、中国政府网、央行/证监会/银保监官网、交易所公告 | 定性、权威表态 |
| P0 | 通讯社 | 路透、彭博、AFP、AP | 海外事件首发 |
| P1 | 头部财经媒体 | 财联社、界面、第一财经、21 经济、证券时报、经济观察报、华尔街见闻 | 事件拓展、市场反应 |
| P1 | 互联网财经 | 36氪、虎嗅、雪球热帖 | 舆论场温度 |
| P2 | 社交/垂类 | 微博热搜、雪球、股吧、小红书财经 | 情绪传导、个体视角 |

**检索动作**：
```
WebSearch 关键词模板：
  - 「[事件主体] [动作词] 最新」       # 抓最新进展
  - 「[事件主体] 官方回应 / 表态」      # 抓定性
  - 「[事件主体] 影响 / 传导」          # 抓解读
  - 「[时间锚] [主题] 深度」            # 抓已有稿件，避免重复
```

**检索完必须回答**（内部自查清单）：
- [ ] 事件**最新发生时间**是什么时候？（精确到时/分）
- [ ] **涉事主体**叫什么、什么层级？
- [ ] **官方表态**有没有？出自谁？原话是？
- [ ] **关键节点日历**（未来 24/48/72 小时有什么重要会议/发布/窗口）？
- [ ] 是否已有**对立叙事 / 分歧观点**（决定要不要做多视角稿）？

---

### 档位 2｜行情数据检索（finance-data 路线）

**什么时候必跑**：
- 稿件涉及任何具体涨跌幅、价格、成交量、市值、资金流向
- 涉及板块轮动、个股动态、ETF 资金流、北向资金
- 涉及指数点位、汇率报价、商品价格、利率
- 涉及历史对比（如"相比上次议息后 A 股怎么走"）

**检索路径优先级**（严格按此顺序，**不要跳过**）：

```
┌─────────────────────────────────────────────────┐
│ 第一优先 · neodata-financial-search           │
│  - 自然语言查询                                 │
│  - 实时数据                                     │
│  - 覆盖 A股/港股/美股/基金/指数/板块/宏观/外汇  │
│  - 默认所有行情类查询走这个                    │
└─────────────────────────────────────────────────┘
              │  若不覆盖 / 字段不足 / 需历史批量
              ↓
┌─────────────────────────────────────────────────┐
│ 第二优先 · finance-data-retrieval              │
│  - 209 个结构化 API                             │
│  - 适合：历史日线、批量、精确字段、复杂约束    │
└─────────────────────────────────────────────────┘
              │  以上都无法满足
              ↓
┌─────────────────────────────────────────────────┐
│ 第三优先 · WebSearch                            │
│  - 明确告知用户"数据非实时"                    │
│  - 必标来源与截止时间                           │
└─────────────────────────────────────────────────┘
```

**检索完必须回答**：
- [ ] 所有稿内硬数字是否**都能追溯到实际调用返回**？
- [ ] 每个数字后面是否都带**"截至 X 时"时间戳**？
- [ ] 是否做了**同比/环比/历史均值**等对比锚点（避免孤立数字）？
- [ ] 是否区分了**盘中/收盘/盘后**数据口径？

**严禁动作**（红线）：
- ❌ 凭记忆写具体价格（"当前大概 3.2 万点"）
- ❌ 凭估算写涨跌幅（"差不多涨了 2 个点"）
- ❌ 用过时数据装最新（超过 24 小时的行情数据不标时间）
- ❌ 用港股数据冒充 A 股、用期货冒充现货

---

### 档位 3｜宏观/监管/行业数据（多源组合）

**什么时候必跑**：
- 涉及 GDP / CPI / PPI / PMI / 社融 / M2 / 失业率 等宏观指标
- 涉及政策文件、监管新规、行业标准
- 涉及行业规模、产业链、竞争格局数据

**检索清单**：
| 数据类型 | 首选信源 | 备选 |
|---|---|---|
| 宏观指标 | 国家统计局官网、央行官网 | `neodata` → `cn_gdp` / `cn_cpi` / `shibor` 等 API |
| 政策文件 | 国务院客户端、中国政府网、各部委官网 | 新华社 / 央视新闻通稿 |
| 监管文件 | 证监会 / 银保监 / 人行官网 | 财联社监管快讯 |
| 行业数据 | 工信部 / 协会 / 官方白皮书 | 券商研报摘要（需备注二次引用）|
| 海外宏观 | Fed / ECB / BIS / IMF 官网 | 路透 / 彭博 |

**检索完必须回答**：
- [ ] 是否取到**最新一期**（不是去年/上季度的数据）？
- [ ] 口径是否统一（同比 vs 环比、现价 vs 不变价、季调 vs 未季调）？
- [ ] 来源是否**一手**（政府/央行/协会），还是二手引用？

## 🧭 超级事件数据包（evaluation ≥ 40 的标配）

当事件能量到超级级别，**三档都要跑齐**，组合成一个"数据仪表盘"：

```
【事件事实】
  - 时间线（最近 24h 主要节点）
  - 官方表态（美方/我方/欧方/当事方）
  - 下一关键窗口（24/48/72h）

【行情读数】
  - 核心品种（油/金/汇率/主要指数）即时行情
  - 板块表现（A 股涨跌前 5 板块 + 领涨领跌个股）
  - 资金流向（北向、融资融券、ETF 申赎）
  - 衍生品（VIX、期权 PCR、期货基差）

【宏观背景】
  - 近期相关政策文件
  - 历次类似事件的宏观数据对比
  - 权威研报预测区间
```

**这个仪表盘不需要全部写进稿，但必须全部查过**——查了才知道哪些写进稿最有价值。

## ⏱️ 数据检索时效标准

| 数据类型 | 必须多新 | 超时处理 |
|---|---|---|
| 事件事实 | **过去 6 小时内** | 超时必须重新检索 |
| 盘中行情 | **过去 15 分钟内** | 超时必须重新调用 |
| 收盘数据 | 最近一个交易日 | 隔日必须更新 |
| 宏观月度 | 最新一期 | 标注发布日期 |
| 宏观季度 | 最新一期 | 标注发布日期 |
| 政策文件 | 最新版 | 注明文号与生效日 |

**时效原则**：**数据越新越好，但宁可标"截至 X 时"也不要假装"实时"**。

## 🚫 数据缺失如何处理（透明兜底）

当检索不到或结果不可靠时，**严禁虚构**。标准话术：

```
Agent（正确做法）：
  「事件 X 的 [具体数据] 暂未从 [信源] 获取到，
  建议你以 [权威渠道] 的口径为准，
  稿件中该位置我已用占位符标记，待你补充后重校一遍。」

Agent（错误做法）：
  ❌ 凭印象填一个大概数字
  ❌ 写"据悉""大约""有消息称"蒙混过关
  ❌ 用去年同期数据冒充今年
```

**占位符规范**：
- `{{待补：XX 数据}}` → 明确告知用户这里需要补
- `{{估算：XX}}` → 如果确实只能估，必须用 `估算：` 前缀
- 文末列"数据缺口清单"，告诉用户一共还有几处待补

## 🎭 演示/骨架稿的例外规则

当用户明确说：
- "这是演示稿"
- "给我看下骨架就行"
- "不要联网，凭你现有知识写"
- "模板演示，数据占位即可"

→ Agent 可以**跳过数据检索**，但**必须满足两个条件**：

1. **稿件顶部标注**：
   ```
   > ⚠️ 本稿为 skill 形态演示，所有时间/数据/表态均为示例占位，
   > 不作为真实新闻发布依据
   ```

2. **稿件底部重复标注**：
   ```
   > ⚠️ 演示稿提醒：本稿数据为示例占位，正式用稿前所有数字、
   > 人名、机构表态必须以当日通讯社核实信息替换
   ```

这是**双重兜底**，防止用户一转眼把演示稿发出去。

## 📋 Agent 自查输出模板

每次进入出稿流程前，Agent 在内部（可选显式告诉用户）先过一遍：

```
【数据先行自查 · Step 1.5】

事件事实：
  ✅ 已 WebSearch，命中 N 家权威信源（新华社 + 财联社 + 路透）
  ✅ 最新进展时间：XX-XX 11:15
  ⚠️ 官方表态暂只有财政部，其他部门未回应 → 稿中体现"等待多部委表态"

行情数据：
  ✅ 已调 neodata-financial-search：A 股 / 商品 / 汇率 即时读数
  ✅ 已调 finance-data-retrieval：近 3 次相似事件后的板块轮动历史
  - 无数据缺口

宏观背景：
  ⚠️ 最新 PMI 数据为上月，本月数据 4 月 30 日发布 → 稿中标注

→ 数据齐备度：85%，可进 Step 2（评估）
```

## 🎯 给 Agent 的心法

1. **动笔的冲动 = 应该先查的信号**：越想马上开写，越说明你心里没底，越该去查。
2. **查得少 = 编的多**：稿件里"据悉""相关数据显示""有业内人士透露"越多，越说明数据功课没做足。
3. **把自己当记者**：记者写稿前要打电话核实、要拉数据、要查文件——你也要。
4. **数据缺口透明化**：不怕告诉用户"这里我没查到"，怕的是用户发了稿之后才发现你编的。
5. **时效 > 丰富度**：宁可只有 3 个最新数据，不要堆 10 个过时数据。
6. **反推检索边界**：准备写 `A 股 T+24h 板块传导`，就必须至少拉到"历次类似事件后的板块轮动数据"——想清楚稿要写什么，就知道该查什么。
7. **演示稿标注不是礼貌，是硬规则**：骨架稿跳过数据是被允许的，跳过标注是不被允许的。
