# WeStock Brokerage — 券商观点聚合与分析

支持 9 种回答模式，对应 9 种券商观点场景。

---

## 前置步骤：Token 配置

本 Skill 通过 MCP 协议请求 SNP 后台数据（type=3），需要 `SNP_MCP_TOKEN` 进行身份认证。

**每次使用前，先检查 `<skill-path>/.env` 文件**：

1. 读取 `<skill-path>/.env`，查看是否存在 `SNP_MCP_TOKEN=xxx`
2. 如果文件不存在或为空，**默认自动写入环境变量中的 Token**：

```bash
echo "SNP_MCP_TOKEN=Bearer $TAI_IT_TOKEN" > <skill-path>/.env
```

3. 写入后发起一次 MCP 请求验证 Token 是否有效
4. **如果认证不通过（返回权限错误）**，则向用户提示：

> 当前 Token 认证失败。请访问 `https://tai.it.woa.com/user/pat` 获取您的个人令牌（令牌名称：**openclaw-internal-auto**），获取后重新写入 `.env`：
> ```bash
> echo "SNP_MCP_TOKEN=用户提供的token" > <skill-path>/.env
> ```
> ⚠️ 注意：配置的 Token 必须包含 **Bearer** 前缀，否则权限校验不通过。

> Token 仅需配置一次，后续自动读取。运行 CLI 时通过 `--env-file` 自动加载。

**环境要求**：Node.js >= v18

**运行方式**：
```bash
node <skill-path>/scripts/index.js <command> [options]
```

## 执行流程

> ⚠️ **静默执行原则**：以下所有步骤（模式识别、参数提取、CLI 调用、AI 筛选、读模板、列提纲、自检）均为**内部处理过程，禁止向用户输出任何中间过程信息**。禁止出现以下类型的表述：
> - "按 XX 模板"、"正在读模板"、"使用 modeX 模式"
> - "需要查询 SNP MCP"、"调用 CLI 采集数据"
> - "筛选 Top 15 篇文章"、"从 50 篇中筛选"
> - "列提纲"、"自检通过"、提纲内容本身
> - 任何涉及执行步骤、工具调用、内部计数的描述
>
> **用户只能看到最终结果（Step 4.3 的输出内容），其他一切均为静默执行。**

```
用户问题 → 模式识别 → 参数提取 → CLI 采集(仅1轮) → AI筛选Top15 → 读模板 → 提纲 → 输出 → 自检
```

### Step 1 — 模式识别 & 参数提取

从用户问题中提取关键词传入 CLI：

| 优先级 | 规则 |
|-------|------|
| **P0** | 从用户原文提取核心词汇作为 `-t`(title_keyword) 或 `-c`(content_keyword) |
| ⚠️ | **关键词类型限制**：模式 **1/2/5/9** 只传 `-t`（标题关键词），不传 `-c`；其余模式（3/4/6/7/8）**优先传 `-t` 查询，当返回结果较少时再追加 `-c` 补查一轮**（两轮结果合并去重后进入 AI 筛选） |
| P1 | 仅当无法从原文提取有效关键词时，才使用下表中该模式的「兜底关键词」 |
| ⚠️ | **禁止每次都带兜底关键词**；提取的关键词超过 3 个时只取核心 3 个 |
| ⚠️ | 若用户指定了特定券商才传 `-m`(media_keyword)，否则不传 |
| ⚠️ | **金股模式(`gold-stock`)例外**：执行**固定 2 轮查询（不超过 2 轮，禁止换词多次 `-c` 查询）**：
> - **第1轮**：从以下 5 个固定关键词中**任选 2 个**作为 `-c` 参数：「重点推荐」、「持续推荐」、「坚定推荐」、「核心推荐」、「月度精选」
> - **第2轮（仅当第 1 轮返回 < 5 篇时触发）**：改用 **`-t` "金股"** 补查 1 轮
> - ⚠️ 禁止轮换不同关键词组合多次用 `-c` 查询，总共最多查 2 轮
### Step 2 — CLI 数据采集

```bash
westock-brokerage search --mode <mode> [-t "关键词"] [-c "关键词"] [-d <天数>] [-m "券商"]
```

统一调用 MCP `get_common_news`（type=3），参数说明：

| 参数 | 说明 |
|---|---|
| `--mode` | 必传，见下方模式速查表 |
| `-t` / `-c` | 标题/内容关键词（Step 1 提取） |
| `-m` | 来源券商（未指定不传） |
| `-d` | 天数，使用**自然日边界**（0点起算） |
| `--begin_time` / `--end_time` | 秒级时间戳（通常用 `-d` 即可） |

### Step 3 — AI 筛选

后台最多返回 **50 篇**，AI 从中筛选 **Top 15**（按各模式筛选规则打分排序），精读后输出。

> ⚠️ **禁止二次过滤**：返回结果已是后台匹配后的结果，不要再根据标题/内容做额外过滤，直接使用。

### Step 4 — 上下文隔离 → 读模板 → 列提纲 → 输出 → 自检

> ⚠️ **每次回答都必须独立执行完整 Step 4，禁止复用上一次的模板结构或提纲！**

#### 4.0 — 上下文隔离（必做）

在读取模板之前，**必须先明确当前模式标识并清空残留**：

1. **强制输出模板路径**（防止查表出错）：在内声明本次要读取的**完整文件路径**
   - 格式：`📎 本次模板 = <skill-path>/references/<具体文件名>`
   - ⚠️ 必须根据 Step 1 识别的模式从下表选取，**禁止凭记忆填写文件名**

   | 模式标识 | 模板文件路径 |
   |---------|------------|
   | `weekly-strategy` | `<skill-path>/references/mode1-weekly-strategy.md` |
   | `morning-meeting` | `<skill-path>/references/mode1-weekly-strategy.md` |
   | `market-outlook` | `<skill-path>/references/mode2-market-outlook.md` |
   | `sector-allocation` | `<skill-path>/references/mode3-sector-allocation.md` |
   | `sector-opinion` | `<skill-path>/references/mode4-sector-opinion.md` |
   | `sector-compare` | `<skill-path>/references/mode5-sector-compare.md` |
   | `gold-stock` | `<skill-path>/references/mode6-gold-stock.md` |
   | `risk-warning` | `<skill-path>/references/mode7-risk-warning.md` |
   | `stock-opinion` | `<skill-path>/references/mode8-stock-opinion.md` |

2. **清空上下文残留**：
   - 明确告知自己：**上一次使用的模式（如有）与本次不同，必须完全丢弃上一次的模板结构、提纲、输出格式**
   - 常见污染示例（必须避免）：
     - ❌ 上次用了 mode1 的总览+机构共识+券商表格 → 这次 mode2 也用了同样的结构（mode2 要求的是 Part A 表格 + Part B 四维分析）
     - ❌ 上次表格列名是 `大盘观点 \| 行业配置` → 这次直接沿用而没检查当前模式要求的列名
     - ❌ 上次有某个 section → 这次无脑照搬

#### 4.1 — 读模板

1. **读** `<skill-path>/references/output-spec.md`（通用规范：中文标点、第三人称、柱状图等格式要求）
2. **读** Step 4.0 中声明的**具体模式模板文件**（即 `📎 本次模板 =` 指向的文件）
   - ⚠️ 必须使用 4.0 中输出的完整路径，**禁止凭记忆或查速查表**

#### 4.2 — 列提纲（含模式标识锚定）

在内部展开完整输出结构，**提纲头部必须显式标注当前模式**：

```
=== 输出提纲 [模式: XXX] ===
Section 1: <标题> → 字段: ...
Section 2: <标题> → 字段: ...
特殊格式: 柱状图 / 逻辑链 / emoji 等
字数限制: ...
```

> 📌 提纲中必须包含：所有一级标题 → 各字段/表格列名 → 特殊格式（如柱状图、箭头逻辑链 `A→B→C`）

#### 4.3 — 填充内容

严格按提纲逐段写，**每写一段前确认该 section 属于当前模式的模板**。

#### 4.4 — 自检

生成完毕后核对清单再输出：

- [ ] 本次使用的模式标识 = `<当前模式>`（与 Step 1 识别结果一致）
- [ ] 包含模板要求的**所有一级 section**（不多不少）
- [ ] 表格列名与**当前模式**模板完全一致（不是上次的）
- [ ] 特殊格式（柱状图/逻辑链/emoji）齐全
- [ ] 全部中文标点（`%`/`.`/$`除外）
- [ ] 无第一人称表述
- [ ] 字数符合**当前模式**模板限制
- [ ] ❌ **未混入其他模式的框架结构**（如 mode2 的四维框架出现在 mode3 中）

> ❌ **禁止不读模板直接输出！❌ 禁止自由发挥格式！❌ 禁止跳过自检！❌ 禁止复用上次模板！**

#### 4.5 — 典型污染案例（警示）

以下为真实发生的模板套用错误，**必须引以为戒**：

| # | 当前应使用模式 | 错误行为 | 正确做法 |
|---|-------------|---------|---------|
| 1 | **mode2** (`market-outlook`) — 大盘走势 | 输出了 mode1 的「总览段落 + 机构共识逻辑链 + 券商\|大盘观点\|行业配置 表格」结构 | 应读 `mode2-market-outlook.md`：Part A 表格（券商\|大盘核心判断\|指数关键点位\|核心逻辑\|白话总结）+ Part B 四维分析（大势判断\|核心影响因素\|操作建议\|一致看好主线） |
| 2 | **mode3** (`sector-allocation`) — 行业配置 | 沿用了上次的表格列名 `大盘观点 \| 行业配置` | 应读 `mode3-sector-allocation.md`：表格列名必须是 `行业配置建议 \| 看好方向 \| 核心理由` |
| 3 | **任意模式** | 上次回答有某个 section（如"一致看好主线"），这次无脑照搬 | 每次必须重新读模板，只保留当前模板要求的 section |

**根因总结**：AI 在多次调用同一 skill 后会产生「上下文惯性」，倾向复用已熟悉的旧结构而非读新模板。**Step 4.0 的路径声明 + Step 4.1 的强制读取 + Step 4.4 的自检 + 本节的负样本对照**，四道防线共同防止此问题。

### 匹配规则说明

- **标题或内文匹配**：模式 1/2/3/4/6/7/8 — 关键词出现在标题或正文中即命中
- **标题必须匹配**：模式 5/9 — 关键词必须出现在标题中
- **正文必须匹配**：模式 7 — "金股"必须出现在正文中

## 触发条件 & 异常处理

| 场景 | 处理方式 |
|------|---------|
| ✅ **必触发** | 券商策略/晨会/金股、券商怎么看/建议/观点/看好/风险提示、具体券商名 xx 证券 |
| ❌ **不触发** | 纯股票行情查询(用 stock-data)、历史数据/历史观点 |
| ❓ 未指定板块/个股 | 反问："您想了解哪个板块/个股的观点？" |
| ❓ 时间不明确("最近") | 默认查询最近一周 |
| ❓ 指定了券商但问题不完整 | 返回模式3(大盘观点)结果，并追问具体想了解什么 |
| ⚠️ **SNP 无数据** | SNP 查询返回 **0 条**时，在 https://www.hibor.com.cn 抓取数据， 搜索关键词抓取券商研报/观点，按原模式模板输出并标注「*数据来源：汇博网(hibor.com.cn)*」；hibor 也无数据则告知用户暂未检索到 |
